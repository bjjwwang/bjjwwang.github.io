WhyFlow: Interrogative Debugger for Sensemaking Taint Analysis
Burak Yetiştiren · Hong Jin Kang · Miryung Kim  |  UCLA / Univ. of Sydney
ICSE 2026 Research Track arXiv: 2508.07198 Taint Analysis · Debugging CodeQL · Datalog
核心问题 — 为什么要做这个

现有 taint analysis 工具（如 CodeQL）只告诉你"有哪些污点流"，却不告诉你为什么有、为什么没有、改个库的配置会怎样。开发者排查一条 taint flow 告警需要手工沿着 call graph 追踪，极度耗时。

核心 Gap
CodeQL 只给你一棵"告警树"，你要自己猜：这条流为什么出现？哪个第三方库 API 在中间传导？如果我修改了那个库的 model，还有多少流受影响？
系统架构（三层）
┌──────────────┐ 用户写的 Java/QL 代码 │ CodeQL │──── 运行两条 query ────┐ │ (Analysis) │ │ ① 标准 taint query（加库假设） └──────────────┘ │ ② Maximal query（忽略所有库假设） ▼ ┌───────────────────────┐ │ Fact Conversion Layer │ CodeQL 结果 → Soufflé 事实集 │ node/edge/source/sink │ │ plausible_edge │ "如果某个库 model 变了"的潜在边 │ library_flow │ 第三方 API 的传导关系 └───────────────────────┘ ▼ ┌───────────────────────┐ │ Soufflé Engine │ Datalog 推理引擎 │ 6种查询模板（Datalog）│ 5 秒响应 vs CodeQL 10-14 秒 └───────────────────────┘ ▼ ┌───────────────────────┐ │ Interactive UI │ 颜色编码图 + 下拉框 + 点击跳源码 └───────────────────────┘
核心设计 — 6 种交互式查询
查询类型自然语言意图Datalog 实现输出
WhyFlow 为什么 source X → sink Y 有污点流？ 对 edge 作传递闭包，交集 library_flow，定位哪些第三方 API 是传导媒介 绿=source，红=sink，橙=关键 API，实线边
WhyNotFlow 为什么 X → Y 没有流？ 找断掉的可达路径 + plausible_edge（被 sanitizer 封住的潜在边） 虚线边 = 如果改配置则激活的路径
AffectedSinks 如果库 Z 的 model 改变，哪些 sink 会变成不可达？ What-if：移除 library_flow(Z) 后重算可达 sink 集合 受影响 sink 列表
DivergentSinks 哪个库 model 让 source X 分叉到多个 sink？ 找公共分叉节点（下游 sink 集合 > 1 的 branch point） 关键分叉 API，可同时阻断多条流
DivergentSources 哪个库 model 汇聚了多个 source 到 sink Y？ 找公共汇合节点（上游 source 集合 > 1 的 merge point） 高风险汇聚点
GlobalImpact 哪个库 model 影响最多条 X→Y 路径？ 对每个 library_flow node 统计其出现在多少条不同路径中，按频次排序 节点大小正比于全局影响力
关键实现细节
1
双 Query 策略（消除重新分析开销）
一次性同时运行标准 query 和 maximal query，把两个结果的差集存为 plausible_edge 事实，后续所有 why-not/what-if 查询都在预计算事实上做 Datalog 推理，不再调用 CodeQL（避免 10-14s 延迟→变成 5s）。
2
Soufflé Datalog — 交互式查询引擎
所有 6 种查询都是独立的 Datalog 规则文件，直接加载 CodeQL 导出的事实库（.facts 文件）。新增查询 = 写几行 Datalog 规则（作者实测新加一条 divergent path 查询只需 <10 分钟开发）。
3
UI 视觉编码
实线边=当前活跃流；虚线边=受 sanitizer 封锁但 plausible 的流。节点颜色：绿=source，红=sink，橙=library API，蓝=中间语句。GlobalImpact 节点大小 ∝ 影响路径数。

MINES: Explainable Anomaly Detection through Web API Invariant Inference
ICSE 2026 Research Track
ICSE 2026 Research Track arXiv: 2512.06906 Invariant Inference · Anomaly Detection LLM + Schema · Python
核心问题

Web 应用运行时异常检测（攻击、权限越界等）依赖日志分析，但日志嘈杂、不完整：很多关键信息在数据库里，不在 HTTP 日志里。现有方法（LogRobust、LogFormer）学的是日志 pattern，遇到不在训练分布里的新攻击就失效，且假阳性多。

核心洞察
与其从日志"学"什么是正常，不如从代码结构（API 签名 + 数据库 Schema）"推断"应用不变量（invariants），再用这些不变量去校验运行时日志。Schema 是静态的、稳定的，不依赖训练样本分布。

各步骤详解
1
API 签名 → 表结构
把每个 API 端点的名称当表名，参数当列名，嵌套对象展平到指定深度阈值。额外加一个 Env 实体（session/userID/role）——这是普通日志分析遗漏的关键上下文。
2
三类关系推断（关键步骤）
API-DB 关系：对每对 (API参数, DB列) 问 LLM：它们语义上是同一个字段吗？用属性值重叠率做 heuristic 过滤。
API-API 关系：用 HMM（隐马尔可夫模型）在调用日志上学 API 调用时序，找出"A 调用后必须跟 B"这类依赖，低于概率阈值的丢弃。
API-Env 关系：通过 session ID 匹配，确认哪些 API 参数与当前会话用户上下文绑定。
3
Table Join → 联合视图
根据推断出的关系做 left outer join，生成一张"超级宽表"，例如 cancelOrder-orders-Env 把 API 调用参数、被操作的 DB 记录、用户 session 信息合并到一行。
4
LLM 生成候选不变量（5类）
输入：联合表 Schema（不是数据！Token 消耗从 2.4×10⁵ 降到 5×10³）。LLM 输出覆盖：①常识约束（price>0）②格式约束（email格式）③DB 约束（orderId 必须存在于 order 表）④环境约束（userId 与 session 一致）⑤API 时序约束（退款前必须有支付记录）。
5
迭代精化（LLM 对话反馈）
把候选不变量跑在正常训练日志上，违反了的 invariant 就是假的，把违反情况和错误上下文喂回 LLM 同一对话线程，请它修改或丢弃。循环至收敛或达最大迭代次数。精化后假阳性率从 19.7% → 0%。
6
Python 代码生成 + 离线校验
最终 invariant 被编译成可执行 Python 函数，运行时结合 API 日志 + 数据库 Binary Log 回放（这是访问"历史 DB 快照"的关键）来校验每条 API 调用。违反不变量 → 异常告警。

复现关键步骤

必备输入

  • API 签名（OpenAPI/Swagger 规范）
  • 数据库 Schema（DDL 文件）
  • 正常运行时 API 日志（训练集）
  • 数据库 Binary Log（MySQL binlog 等）

最难的坑

  • HMM 学 API-API 时序：数据量需足够
  • DB Binary Log 回放：需对齐时间戳

MINES = 从 API Schema + DB Schema 构建 ER 图 → LLM 推断跨表约束（不变量）→ 正常日志验证精化 → Python 代码化 → 运行时 DB binlog 回放 + 日志联合校验。无需标注攻击样本，零假阳性。

Reducing False Positives in Static Bug Detection with LLMs: An Empirical Study in Industry
Tencent & 合作高校  |  工业实证研究
ICSE 2026 SEIP Track arXiv: 2601.18844 LLM × Static Analysis · FP Reduction GPT-4o · Claude Opus · DeepSeek · Qwen
核心问题

静态分析工具（SAT）的假阳性（FP）率在工业代码上极高，开发者每条告警平均花 10-20 分钟手工核实。腾讯广告系统有大量代码、持续 CI 集成，FP 带来的人工成本不可接受。

学术界已有 LLM 减少 FP 的工作，但全在开源小型代码库上做，工业大规模真实场景下效果未知。这是第一篇工业级实证研究。

Rusted Types: Static Detection of Rust Type Confusion Bugs
Zeyang Zhuang · Wei Meng · Michael Lyu  |  香港中文大学
ICSE 2026 Research Track Rust · Static Analysis · Memory Safety Dataflow · Interprocedural
核心问题

Rust 以内存安全著称，但 unsafe 代码块中的类型混淆（type confusion）漏洞绕过了 Rust 的类型系统保护：开发者用 as、transmute、from_raw 等将一种类型强转为另一种，如果类型的对齐/布局/生命周期不匹配，就会触发内存安全问题。

为什么已有工具不够
C/C++ 的类型混淆检测工具不适用于 Rust（所有权语义完全不同）。Rust 官方类型检查器（rustc）在 unsafe 块中主动放弃了精确类型验证。RustSec CVE 数据库中此类 bug 长期被忽视。
Rust Type Confusion 三大类（来自 TypePulse，同领域参考）
I
Type I — Misalignment（对齐违规）
将 *u8（1字节对齐）强转为 *u32（4字节对齐）。如果地址不满足目标类型对齐要求，引发 UB（Undefined Behavior）。
II
Type II — Inconsistent Layout（布局不一致）
Rust 默认 struct 字段顺序由编译器决定（非 #[repr(C)]），两次编译字段顺序可能不同。跨类型转换时字段语义错位，导致数据损坏。
III
Type III — Mismatched Scope（别名违规）
通过 transmute 创建无效 bit pattern，或把 &T 转成 &mut T 违反 Rust 独占引用规则，导致两个可变指针指向同一内存。
DeRust 技术方案
1
基于 MIR 的跨函数数据流分析
在 Rust 编译器的 MIR（Mid-level Intermediate Representation）层工作，MIR 已完成借用检查但保留了类型信息。追踪从 unsafe 类型转换操作（transmute/as/from_raw_parts）到"敏感操作"（解引用/字段访问/返回值传播）的跨函数数据流。
2
类型空间分析（对齐/布局/可变性）
对每对转换类型 (T_src, T_dst) 静态计算：①对齐要求是否满足，②是否涉及非 repr(C) struct，③可变性约束是否被破坏。
3
泛型处理（trait bound 解析）
Rust 泛型导致转换的具体类型在编译时不定，工具递归解析 trait bound，枚举所有满足约束的具体类型，对每种具体类型做检查。

TypePulse: Detecting Type Confusion Bugs in Rust Programs
Hung-Mao Chen et al.  |  同领域参考工作，与 DeRust 相互印证
USENIX Security 2025 arXiv: 2502.03271 Rust · Property Graph · Alias Analysis MIR · Datalog
为什么放这里
TypePulse 与 ICSE 2026 的 DeRust 解决同一问题（Rust type confusion），但发表于 USENIX Security 2025，技术方案更完整公开（arXiv 全文可读）。两篇论文互为对照，读懂一篇对理解另一篇极有帮助。
系统架构
Rust 源码 ↓ rustc 编译 MIR (Mid-level IR) ↓ Property Graph Constructor ├── 类型转换分析: 收集所有 unsafe 转换对 (T_src, T_dst) + trait bounds └── 指针别名分析: 构建有向别名图（所有权转移 = 断边） ↓ Property Graph: 类型对 + 别名图 + 函数元数据 统一图结构 ↓ Bug Detector ├── Type I 检测: 对齐整除性检查 ├── Type II 检测: 非 stable layout 转换识别 └── Type III 检测: 弱/强类型转换 + 可变性约束 ↓ Bug Reports (71 new bugs in top-3000 packages)
指针别名图的核心设计（Algorithm 2）

TypePulse 构建的别名图完全遵循 Rust 所有权语义，而不是 C/C++ 的传统别名分析：

①
Ref / RawPtr 语句 → 添加边 dst → src
let y = &x 或 let p = x as *const T：y 指向 x，添加有向边。
②
PtrToPtr / Transmute → 保持别名关系
类型转换后两个指针仍指向同一内存，别名关系保留（这是 type confusion bug 的根源！）。
③
Move 操作数 → 断开原有向边
所有权转移后原变量不可用，断开其出边——这是区别于 C 别名分析的关键，避免追踪已失效的指针。
④
StorageDead → 删除所有出边
变量生命周期结束，删除其所有别名出边，精确反映 Rust 生命周期语义。
